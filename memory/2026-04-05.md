### 4. OpenClaw Architecture Research

**Web search results analyzed:**

**Alex Finn's Videos (YouTube):**
- "OpenClaw is 100x better with this tool (Mission Control)"
- Mission Control = dashboard for agent visibility
- Focus: Kanban + real-time updates + budget alerts

**Community consensus (honest take):**
1. **Dashboard ≠ Value** — Kanban board looks impressive but actual value is: scheduled tasks, memory, persistent context
2. **Alex Finn's setup** — visual dashboard, but he's also spending 4-5am every night hacking (not sustainable for average users)
3. **Dan Malone** — tried dashboard, abandoned it! Switched to just Telegram + OpenClaw. "The gap wasn't coordination UI. It was persistence + mobile access + cross-agent collaboration."
4. **What actually matters:**
   - ✅ Agent always-on (24/7 uptime)
   - ✅ Phone access (Telegram/WhatsApp)
   - ✅ Agents can communicate with each other
   - ✅ Persistent memory across sessions
   - ❌ Dashboard is **optional** — uptime is **mandatory**

**OpenClaw Agent Communication (from Medium article by Chen Yang):**
- Two mechanisms: `sessions_spawn` (spawn tasks to other agents) and `sessions_send` (direct messaging)
- Requires config: `tools.agentToAgent.enabled = true`
- Max 5 ping-pong turns per session
- Example: Megan (main) → Todd (Product Owner) → Alex (Backend Dev)
- Each agent needs separate Discord bot, shared channel access, bindings

**How OpenClaw Actually Works (from openclaw.design + dev.to):**
- 5 input types: Messages, Heartbeats (~30 min), Cron Jobs (exact timing), Hooks (internal triggers), Webhooks (external systems)
- Memory = Markdown files + semantic search (QMD)
- Agent-to-agent communication = sessions_send + sessions_spawn
- Memory retrieval = smart retrieval from files (not neural)
- "Proactive" = heartbeats + cron + webhooks (not thinking between messages)
